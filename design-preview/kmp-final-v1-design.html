Architecture — CBM + KMP Integration

CBM = stateless data generator (free OSS). KMP = stateful knowledge engine (freemium). CTO-approved boundary.

User Guide — How the two tools work together
What it does
Codebase Map (CBM) parses your code into a function dependency graph (nodes + edges). KMP imports that graph, learns patterns, and provides AI-powered Q&A. CBM is the “data generator”, KMP is the “intelligence layer”.
How to use
You don’t need to manage the integration manually. codebase-memory bootstrap handles both CBM graph generation and KMP learning in one pipeline. The impact command internally calls CBM’s dual-snapshot diff engine.
Key principle
CBM is 100% free and open source. KMP adds AI intelligence on top. Lifecycle commands (state management) live in KMP, not CBM. This keeps CBM lean and focused.
┌──────────────────────────────────────────────────────────┐ │ CODEBASE-MAP v2.2 (standalone, free, OSS) │ │ │ │ [AST Parser] → [Graph Builder] → [graph.json] │ │ [SnapshotDiff] → [Affected Callers] → diff.json │ │ [CI: cbm-pr-impact.yml] → Auto PR comment │ └────────────────────────────┘│└─────────────────────────────┘ graph.json + diff.json ▼ ┌──────────────────────────────────────────────────────────┐ │ KMP (stateful knowledge engine, freemium) │ │ │ │ [CodebaseVault] SQLite + patterns + snapshots │ │ [3 Learners] Naming + Layer + GitOwnership │ │ [AI Gateway] Anthropic / OpenAI / Gemini (BYOK) │ │ [RAG Engine] BM25 + AST graph proximity │ │ [MCP Server] 4 tools for Claude/Cursor/AI agents │ │ [CLI] init · bootstrap · summary · ask · why · impact │ └──────────────────────────────────────────────────────────┘
CTO Decision (10/04/2026)
Boundary: CBM = stateless. KMP = stateful. Lifecycle commands belong in KMP M3, NOT CBM.
Integration: KMP calls codebase-map snapshot-diff --format json via subprocess. No library import.
Pricing: CBM 100% free (OSS). KMP freemium (Pro features in M3+).

M0 codebase-memory init

Vault initialization. Creates .knowledge-memory/ directory + SQLite + config.

User Guide — What is this?
What it does
Initializes a Knowledge Memory vault in your project root. Creates SQLite databases, config file, and directory structure that stores learned patterns about your codebase. Run this once per project.
How to use
Run codebase-memory init in your project root. If vault already exists, use --force to reinitialize (warning: destroys existing data). After init, edit .knowledge-memory/config.yaml to customize scan scope.
Result quality
  • Perfect: All 6 checkmarks green, vault.db created, .gitignore updated
  • Good: All checkmarks but .gitignore already had the entry
  • Bad: Error — usually permissions issue or disk full
Next step
Run codebase-memory bootstrap to start learning patterns from your code.

M1 codebase-memory bootstrap

5-step pipeline: Parse → Snapshot → Learn → Commit → Quick Wins. SLA: ≤5 min for 100K LOC.

User Guide — What is this?
What it does
Scans your entire codebase using AST parsing, then runs 6 learners to discover patterns (naming conventions, architecture layers, error handling, DI, testing, ownership). Produces patterns.md + quick-wins.md with actionable insights.
How to use
Run codebase-memory bootstrap after init. Use --resume to continue after Ctrl+C. Runs automatically on first time; re-run to refresh after code changes.
Result quality
  • Perfect: 25+ patterns, all 6 learners pass, <5 min, 10 quick wins generated
  • Good: 15-25 patterns, 5/6 learners (1 skipped), <5 min
  • Normal: 5-15 patterns — small codebase or few conventions established
  • Bad: 0 patterns — check scan scope in config.yaml, ensure Python files exist
Improve results
  • Expand scan scope in config.yaml (include more directories)
  • Ensure git history has 50+ commits for ownership patterns
  • Lower confidence threshold from 60% to 40% for exploratory analysis

M1 Quick Wins — 10 Insights

5 Structure + 3 Patterns + 2 Risks. Deterministic: same codebase = same output.

User Guide — Reading Quick Wins
What it does
Generates 10 actionable insights from your vault patterns. Fixed ratio: 5 architecture structure + 3 naming/pattern + 2 risk items. Each insight includes confidence score and evidence file paths. Saved to quick-wins.md.
How to read
Each insight shows: number, title, description, evidence path, confidence %. Green confidence (≥80%) = high certainty. Yellow (60-79%) = review suggested. Red = risk item (always important regardless of %).
Result quality
  • Perfect: 10 insights, all ≥70% confidence, clear evidence paths
  • Good: 8-10 insights, 2-3 below 70% — codebase has some unconventional areas
  • Normal: 5-7 insights — small project, fewer patterns to detect
  • Bad: <5 insights or all <60% — scan scope too narrow or project too small
Improve results
Focus on risk items first (items 9-10). God classes >30 methods should be split. Single-owner domains need pair programming or knowledge transfer sessions.

M2 codebase-memory ask

Natural language Q&A. RAG retrieves context from vault, LLM generates answer with citations.

User Guide — How to ask questions
What it does
Ask any question about your codebase in natural language. Uses BM25 + graph proximity to find relevant context from vault patterns and function data, then sends to your configured LLM provider for an answer with file:line citations.
How to use
Run codebase-memory ask "How is authentication handled?". Use --stream for real-time output. Requires LLM provider configured in config.yaml (BYOK: your own API key).
Result quality
  • Perfect: 3+ citations, answer references specific files/lines, <5s response
  • Good: 1-2 citations, answer is relevant but generic
  • Normal: 0 citations, answer from LLM knowledge only (vault context not matched)
  • Bad: "Not enough context" — run bootstrap first or rephrase question
Better answers
  • Be specific: "How does auth middleware validate JWT tokens?" > "How does auth work?"
  • Use function names: "What does OrderService.submit_order do?"
  • Re-run bootstrap after code changes for fresh context

M2 codebase-memory why

Explain why two modules are connected. Call graph analysis + pattern references.

User Guide — Understanding dependencies
What it does
Finds all call paths between two functions/modules using BFS on the call graph. Shows direct, transitive, and shared-module paths. Detects cross-domain coupling and references related patterns.
How to use
codebase-memory why "OrderService calls BillingService". Names are fuzzy-matched — partial names work.
Result quality
  • Perfect: Multiple paths found, cross-domain note with suggestion
  • Good: 1 direct path found
  • Bad: "No path found" — names may be wrong, try partial match
Act on results
Cross-domain coupling (e.g., order→billing) often indicates need for shared interface or event bus. Check architecture note in output for specific suggestions.

M2 codebase-memory impact

Change impact analysis. Uses vault call graph + CBM dual-snapshot engine.

User Guide — Before you change code
What it does
Predicts the impact of modifying a function. Finds all callers (direct + transitive), classifies risk (HIGH=route/middleware, MEDIUM=service, LOW=test), and recommends testing strategy. Optionally integrates with CBM dual-snapshot diff.
How to use
codebase-memory impact "auth/service.py:authenticate". Use --depth 2 for transitive callers. Accepts function name or file:function format.
Result quality
  • Perfect: Risk level shown, callers listed with file paths, recommendation given
  • Good: Callers found but risk assessment is MEDIUM (ambiguous impact)
  • Normal: LOW risk — function has few callers, safe to modify
  • Bad: HIGH risk with 10+ route callers — add integration tests before touching
Reduce risk
  • Always add tests for HIGH-risk function callers before modifying
  • Consider extracting shared interface if function has 10+ cross-domain callers
  • Use --depth 3 to see full blast radius before major refactors

M2 LLM Provider Configuration

BYOK multi-LLM. Anthropic first, then OpenAI + Gemini. Factory pattern.

User Guide — Setting up your LLM provider
What it does
KMP supports 3 LLM providers: Anthropic (Claude), OpenAI (GPT), Google (Gemini). BYOK = Bring Your Own Key. Your API key stays in env vars, never stored in code or vault.
How to use
Edit .knowledge-memory/config.yaml → set llm.provider and llm.api_key_env. Then export ANTHROPIC_API_KEY="sk-ant-...". Optional: configure llm.fallback for automatic provider switching on rate limits.
Result quality
  • Perfect: Primary + fallback configured, cost tracking active
  • Good: Single provider configured, working
  • Bad: "No LLM provider configured" — set llm.provider + export API key
Cost tip
Gemini Flash is cheapest ($0.10/1M tokens). Claude Sonnet is best quality/cost balance. Use codebase-memory usage to monitor spending.

Generates a self-contained HTML dashboard showing: pattern confidence heatmap, learner efficiency cards, ROI calculator, and LLM usage summary. Opens in any browser, works offline, no external dependencies.
How to use
codebase-memory insights --html exports to insights.html. Without --html, shows terminal summary only. Share the HTML file with stakeholders or add to team wiki.
Result quality
  • Perfect: Dashboard shows 25+ patterns, positive ROI, usage data from 10+ queries
  • Good: Dashboard renders but ROI section empty (no LLM usage yet)
  • Bad: Empty dashboard — run bootstrap + ask some questions first
Share with team
Export monthly and share with engineering leadership to demonstrate KMP value. ROI section quantifies time saved vs. API cost — perfect for budget justification.

MEM-M3-08/09 — Local-only telemetry + ROI calculator. Privacy-safe.

User Guide — Measuring KMP value
What it does
Calculates return on investment: (hours saved × hourly rate) − LLM token cost. Hours saved estimated from onboarding reports (2.5h each) + ask queries (0.1h each). Default rate: $30/h (configurable). All data stays local — privacy-safe.
Result quality
  • Perfect: ROI > $200, return multiple > 1000x — tool is paying for itself
  • Good: ROI positive, multiple > 100x
  • Normal: ROI near zero — early usage, keep generating reports + asking questions
  • Bad: Negative ROI — high token cost with few reports. Switch to cheaper LLM model
Maximize ROI
  • Generate onboarding reports for every new team member (biggest ROI driver)
  • Use ask for code review prep instead of manual exploration
  • Set hourly_rate in config.yaml to your team’s actual rate for accurate ROI
