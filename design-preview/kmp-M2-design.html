D CBM Dual-Snapshot ↔ KMP Integration Architecture

How CBM v2.2 (stateless graph tool) feeds into KMP (stateful knowledge engine). CTO-approved boundary.
┌──────────────────────────────────────────────────────────┐ │ CODEBASE-MAP v2.2 (standalone, free, OSS) │ │ │ │ [AST Parser] → [Graph Builder] → [graph.json] │ │ │ │ │ │ [SnapshotDiff] ←→ [Affected Callers] ← [Snapshots] │ │ │ │ │ [diff_formatter] → markdown / json / text │ │ │ │ │ [CI: cbm-pr-impact.yml] → Auto PR comment │ └────────────────────────────┘│└─────────────────────────────┘ │ graph.json + diff.json ▼ ┌──────────────────────────────────────────────────────────┐ │ KMP (stateful knowledge engine, freemium) │ │ │ │ [CodebaseVault] ← imports graph.json as corpus │ │ │ │ │ [AI Gateway] → [RAG: BM25 + graph] → [LLM Provider] │ │ │ │ │ [MCP Server] → 4 tools for Claude/Cursor/AI agents │ │ │ │ │ [CLI: ask / why / impact] → answers with citations │ └──────────────────────────────────────────────────────────┘ Principle: CBM = stateless data generator. KMP = stateful intelligence layer. Lifecycle commands (state management) → KMP M3 (not CBM)
CTO architecture decision (10/04/2026)
Boundary rule: CBM generates graph + diff (JSON). KMP imports + enriches with patterns + LLM. No duplication: KMP calls codebase-map snapshot-diff --format json via subprocess, does NOT reimplement diff logic. Lifecycle commands (state management, task tracking) belong in KMP M3, NOT in CBM. CBM v2.2 shipped: SnapshotDiff, affected callers, rename detection, CI PR Impact — all ready for KMP to consume.
PO business value
CBM = free OSS drives adoption. KMP = freemium monetizes intelligence layer. impact command is the bridge: uses CBM data + KMP patterns + LLM reasoning → actionable insight. This combination is the moat — no competitor (Cursor, Aider, Sourcegraph) has function-level structural diff + AI explanation.

E LLM Provider Configuration MEM-M2-01 / M2-09 / M2-10

BYOK (Bring Your Own Key) multi-LLM. Config-driven provider selection. Anthropic first, then OpenAI + Gemini.
.knowledge-memory/config.yaml
# LLM Provider Configuration llm: provider: "anthropic" # anthropic | openai | gemini model: "claude-sonnet-4-20250514" api_key_env: "ANTHROPIC_API_KEY" # env var name max_tokens: 1024 temperature: 0.3 # low for factual answers # Fallback provider (optional) fallback: provider: "openai" model: "gpt-4o-mini" api_key_env: "OPENAI_API_KEY" # RAG Configuration rag: max_chunks: 5 # context window limit bm25_weight: 0.6 # keyword relevance graph_weight: 0.4 # structural proximity min_score: 0.3 # filter low-relevance
core/ai/providers.py — LLMProvider ABC
# HC-AI | ticket: KMP-M2 class LLMProvider(ABC): """Multi-LLM provider abstraction.""" PROVIDER_NAME: str @abstractmethod def chat(self, messages, **kw): """Send chat request, return response.""" ... @abstractmethod def count_tokens(self, text: str) -> int: ... @abstractmethod def estimate_cost(self, inp, out) -> float: ... # Factory def get_provider(config) -> LLMProvider: return { "anthropic": AnthropicProvider, "openai": OpenAIProvider, "gemini": GeminiProvider, }[config["llm"]["provider"]](config)
CTO constraint
BYOK mandatory: No API key stored in code or vault. Only env var name in config. Fallback logic: If primary fails (rate limit, network), auto-switch to fallback provider + show warning. Cost tracking: Every LLM call logged: provider, model, tokens in/out, cost estimate, latency. No vendor lock-in: LLMProvider ABC means adding new provider = 1 new file implementing 3 methods.

Designer note — Phase split
M2a (Week 1, ~16 SP): AI Gateway + RAG + 3 CLI commands. User gets ask/why/impact working. M2b (Week 2, ~12 SP): MCP server + additional providers + E2E. AI agents get tool access. Each phase ends with a PR. PR-per-day rule applies within each phase.
CTO architecture constraints for M2
1. LLM calls are lazy: Only ask/why invoke LLM. impact can work offline via CBM diff engine.
2. Core ↛ Vertical enforced: import-linter CI gate from M0 still active.
3. MCP protocol isolation: Adapter pattern (M2-07) wraps SDK — if MCP spec changes, only adapter file needs update.
4. RAG recall gate: MEM-M2-02 must demonstrate ≥75% recall on 3 test queries before PR merge.
5. Cost transparency: Every LLM call shows token count + cost estimate. No hidden API calls.

D-M2-06 Fallback provider = automatic CTO Rate limits are common. Auto-fallback with warning (not error). User configures fallback in config.yaml.
D-M2-07 Split M2a/M2b for risk management PM M2a delivers user value (CLI commands). M2b extends to AI agents (MCP). If M2b slips, M2a still ships.
